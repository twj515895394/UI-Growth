# 对 ibelick/ui-skills 的借鉴分析

参考项目：[`ibelick/ui-skills`](https://github.com/ibelick/ui-skills)

## 一、结论

该项目可以为 UI Growth 提供方法论和 Skill 组织方式上的参考，但不能直接作为 UI Growth 的产品方向。

`ui-skills` 更像一个面向设计工程师的 UI Skill 注册、分类和路由集合；UI Growth 则是面向缺少 UI 专业语言的开发者，负责把截图和口语意见转换为可执行任务，并持续建立项目规范和验收闭环。

## 二、值得借鉴的部分

### 1. 最小必要上下文

参考项目的 Root Skill 强调按任务选择最小的 Skill 集合，避免一次加载大量无关规则。

UI Growth 的落地：

- 主 Skill 负责完整流程和路由；
- 问题词典、澄清协议、项目审计、规范治理和视觉验收拆入 references；
- 根据任务只加载最相关的参考文件。

### 2. 先识别任务类型再行动

参考项目先判断任务属于系统、视觉、交互、性能等哪个方向。

UI Growth 的落地：

- 先分类视觉问题；
- 再判断修改范围和风险；
- 再决定直接执行、专业确认或规范治理。

### 3. 基于证据生成设计文档

`create-design-md` 强调从已有设计文档、Token、主题、公共组件和代表页面中提取规则，而不是把每个重复值都提升为正式设计意图。

UI Growth 的落地：

- 建立证据台账；
- 区分已确认、候选、例外、冲突、废弃和未知；
- 正式规范只记录有证据或用户确认的规则；
- 审计报告单独保存未确认建议和冲突。

### 4. 截图不能单独证明精确值

参考项目在 URL/页面分析中优先使用 DOM、Computed Style 和已加载样式。

UI Growth 的落地：

- 截图用于理解视觉症状和用户指向；
- 精确值优先从源码、Token、组件属性和 Computed Style 获取；
- 仅凭截图得到的值必须标记为估算或建议。

### 5. 规则必须能改变实现选择

避免“保持美观”“适当留白”等无法执行的空泛表述。

UI Growth 的落地：

- 所有规范和验收条件必须可执行或可观察；
- 用户表达会被转换为对象、问题、范围、目标、约束和验收条件。

## 三、不应照搬的部分

### 1. 不照搬 Skill 市场和 CLI 路由架构

UI Growth 当前是一个聚焦单一问题域的 Skill，不需要先建设大型 Registry、网站和 CLI。未来只有在专项模块明显增多时，才考虑轻量路由器。

### 2. 不照搬特定技术栈强制项

`baseline-ui` 中存在 Tailwind、React 动画库和特定组件原语的强约束。这些规则适合它自己的目标环境，但不适合 UI Growth 的通用定位。

UI Growth 必须适配：

- Vue、React、原生 Web、小程序；
- CSS、SCSS、Less、CSS Modules、Tailwind；
- Element Plus、Ant Design、Arco、Vant、自研组件等。

### 3. 不照搬固定审美偏好

某些基线 Skill 会禁止或偏好特定渐变、阴影、字距和视觉风格。UI Growth 不应该用一套审美覆盖所有产品。

UI Growth 应优先判断：

- 项目既有设计语言；
- 产品类型；
- 用户任务；
- 信息密度；
- 当前问题是否真的来自这些视觉属性。

### 4. 不把“快速清理 AI 味”当作主要目标

UI Growth 的目标不是一次性把页面“变好看”，而是帮助用户准确表达、做出专业决策、控制修改范围、完成验收并积累项目规范。

### 5. 不把规范生成与具体修改割裂

UI Growth 中，规范不是独立文档任务。局部 UI 调整会读取现有规范；系统级修改会更新决策；高价值页面会形成页面规范；验收会检查规范一致性。

## 四、UI Growth 的独特能力

相较参考项目，UI Growth 重点增加：

1. 面向后端开发者的口语 UI 意图翻译；
2. 截图指代和模糊描述解析；
3. 由专业顾问给出建议后再确认；
4. 修改范围 S1–S5 和风险分级；
5. 可交给代码 Agent 的执行合同；
6. 修改前后截图、状态和响应式验收；
7. 项目规范、审计报告和设计决策的持续治理；
8. 每次任务附带少量、场景化 UI 知识。

## 五、后续借鉴原则

未来继续参考外部 UI Skill 时，必须先回答：

1. 是否提升“意图理解 → 执行 → 验收 → 沉淀”闭环；
2. 是否适配多技术栈；
3. 是否基于项目证据；
4. 是否会把 UI Growth 带向通用 Skill 市场或固定审美；
5. 是否能帮助缺少 UI 基础的用户更准确地参与决策。

只有满足核心方向的能力才应纳入。
