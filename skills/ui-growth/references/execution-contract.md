# UI 修改执行合同

执行合同用于把用户口语、截图诊断和专业建议转换为代码 Agent 可以稳定执行的任务边界。

## 一、为什么需要执行合同

UI Agent 常见失败原因：

- 只收到“优化一下”的模糊目标；
- 未区分视觉症状与实现原因；
- 未指定不可修改内容；
- 没有响应式和状态要求；
- 修改范围在执行过程中不断扩大；
- 代码能编译，但页面效果没有被验证。

执行合同让需求在修改前达到“可执行、可验收、可回退”。

## 二、必填字段

### 1. 页面定位

- 项目或前端包；
- 路由；
- 页面名称；
- 视口；
- 目标组件或截图标注编号。

### 2. 用户原始意图

保留用户原话，避免专业翻译后丢失真实目的。

### 3. 专业诊断

包含：

- 主问题分类；
- 次问题分类；
- 当前观测事实；
- 可能根因；
- 证据来源。

### 4. 修改范围

标明 S1–S5：

- 当前元素；
- 当前组件；
- 当前页面；
- 公共组件；
- 全局系统。

### 5. 目标效果

必须是可观察的结果，例如：

- 两列卡片之间的水平间距与项目其他双列页面一致；
- 所有输入框顶部和底部边缘对齐；
- 1280px 下不出现横向滚动；
- 加载前后按钮宽度不发生变化。

不要只写“更美观”“更高级”。

### 6. 推荐值与来源

每个关键值标记为：

- 现有 Token；
- 公共组件默认值；
- 其他代表页面值；
- 浏览器测量值；
- Skill 建议值；
- 用户确认值。

### 7. 不可修改内容

例如：

- 不修改接口调用；
- 不改变字段顺序；
- 不更换组件库；
- 不影响其他页面；
- 不改变移动端交互；
- 不新增依赖。

### 8. 技术约束

- 使用现有样式方案；
- 优先复用 Token；
- 禁止 `!important`；
- 禁止无理由固定高度；
- 不修改无关文件；
- 不重构业务逻辑。

### 9. 响应式要求

说明：

- 项目支持哪些设备；
- 需要检查哪些视口；
- 允许如何换行、堆叠或滚动；
- 哪些内容可以隐藏；
- 哪些操作必须始终可见。

### 10. 状态要求

根据组件检查：

- Default；
- Hover；
- Focus；
- Disabled；
- Loading；
- Empty；
- Error；
- Success。

### 11. 验收条件

使用明确的通过/失败条件。

### 12. 输出要求

要求 Agent 返回：

- 修改文件；
- 修改前后关键值；
- 修改原因；
- 截图；
- 视口；
- 控制台错误；
- 未解决项；
- 是否更新规范。

## 三、执行过程规则

1. 先定位样式来源，再修改；
2. 先做最小修复，再考虑重构；
3. 如果发现根因与任务合同不一致，暂停扩大修改，重新说明；
4. 如果必须修改公共组件，先做影响分析；
5. 每个修改都应能映射到一个验收条件；
6. 未被合同覆盖的“顺手优化”默认不做。

## 四、发现额外问题时

执行中发现其他 UI 问题，应分成：

- **阻塞问题**：不处理就无法完成当前任务，可一并修复并说明；
- **关联问题**：与当前根因直接相关，可提出但不要擅自扩大范围；
- **无关问题**：记录到建议列表，不在本次修改。

## 五、完成标准

同时满足才算完成：

- 代码检查通过；
- 目标效果可观察；
- 相关视口通过；
- 相关状态通过；
- 无新增控制台错误；
- 未破坏已有规范；
- 修改范围与合同一致；
- 报告包含实际证据。
