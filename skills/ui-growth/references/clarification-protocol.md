# 专业澄清与决策协议

UI Growth 的澄清目标不是把设计责任重新丢给用户，而是帮助用户理解真正的决策点，并在必要时作出明确选择。

## 一、什么时候不应该询问

以下情况通常直接执行：

- 用户意图清楚，且只影响当前局部；
- 当前实现明显偏离项目已有规范；
- 文本遮挡、溢出、错位等客观缺陷；
- 同类控件出现意外不一致；
- 修改可以安全回退，且不会改变产品结构；
- 可以从现有 Token、公共组件或相邻页面得到明确答案。

不要因为用户没有提供精确像素值就立即询问。先从项目证据中寻找合理值。

## 二、什么时候应该询问

当不同答案会实质改变以下任一项时，需要确认：

- 修改范围；
- 页面结构；
- 信息优先级；
- 全局组件行为；
- 响应式策略；
- 视觉风格；
- 重要内容是否隐藏或删除；
- 用户任务效率与可读性的取舍。

## 三、询问前必须完成的分析

在询问用户前，先回答：

1. 用户指向的是哪个对象？
2. 当前观测到的事实是什么？
3. 存在哪两种或多种合理解释？
4. 哪个方案最符合项目现状？
5. 每个方案会影响哪些页面或组件？
6. 是否存在低风险默认方案？

如果这些问题尚未分析，不要直接发问。

## 四、一个好问题的结构

每次只问一个真正会改变结果的问题，并包含：

1. **事实**：当前实现是什么；
2. **分歧**：为什么有多种解释；
3. **推荐**：建议采用哪个方案；
4. **理由**：为什么推荐；
5. **选项**：2–4 个具体方案；
6. **影响**：每个方案影响范围。

示例：

```text
当前两个卡片之间的 column-gap 为 32px，而卡片内部 padding 为 20px，后者与项目其他页面一致。

我推荐只把卡片间距改为 16px，这会提升横向信息密度，但不会让卡片内容变拥挤。

请选择：
A. 只缩小卡片之间的间距到 16px（推荐，只影响当前容器）
B. 间距改为 16px，同时把卡片内部 padding 改为 16px（整体更紧凑）
C. 保持现状，只缩小页面左右边距（会改变整个页面的布局密度）
```

## 五、禁止的询问方式

不要问：

- “你想怎么改？”
- “你喜欢什么风格？”
- “要不要紧凑一点？”
- “具体多少像素？”
- “要不要全局修改？”

除非已经先给出专业分析和具体选项。

## 六、风险分级与确认策略

### 低风险

特征：局部、明确、可回退、不影响公共系统。

策略：直接执行，完成后说明。

### 中风险

特征：影响当前页面多个区域、公共组件 Variant、页面密度或响应式表现。

策略：给出推荐方案和一个保守备选，让用户确认。

### 高风险

特征：全局 Token、默认组件行为、主要页面结构、移动端核心交互或整体视觉语言。

策略：必须确认，并说明影响面与迁移成本。

## 七、当用户无法判断时

用户可能回复“我也不知道，你看着办”。此时：

1. 采用最小影响方案；
2. 优先遵循项目已有规范；
3. 保留业务结构和信息完整性；
4. 选择容易回退的局部修改；
5. 在结果中说明这是“专业推荐方案”，不是用户确认的全局规则。

## 八、确认结果的记录

涉及 S4 公共组件级或 S5 全局系统级决策时，应记录到：

`docs/ui/UI-DECISION-LOG.md`

记录内容：

- 日期；
- 决策；
- 背景问题；
- 选择方案；
- 被否决方案；
- 影响范围；
- 是否需要迁移旧页面；
- 用户确认状态。
