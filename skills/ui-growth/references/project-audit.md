# 项目 UI 现状审计

用于在生成规范、判断局部问题或评估公共组件影响前，提取项目中已经存在的 UI 证据。

## 一、审计目标

不是统计所有 CSS 数值，也不是把历史实现全部升级为标准，而是回答：

1. 项目使用什么技术栈和组件体系；
2. 哪些规则是明确的公共约定；
3. 哪些模式在多个代表页面稳定复用；
4. 哪些实现彼此冲突；
5. 哪些只是局部例外或历史遗留；
6. 哪些决策需要用户确认。

## 二、扫描顺序

### 1. 项目说明和现有规范

查找：

- `AGENTS.md`；
- `README.md`；
- `DESIGN.md`；
- `docs/ui/`；
- 前端开发规范；
- 组件库使用说明；
- 主题和品牌说明。

现有明确规范的优先级高于从代码中推断的模式。

### 2. 技术栈

确认：

- React、Vue、Angular、原生 Web、小程序或其他；
- CSS、SCSS、Less、CSS Modules、CSS-in-JS、Tailwind；
- 组件库；
- 路由结构；
- 构建工具；
- Storybook、测试或截图工具；
- 是否存在多端或多主题。

### 3. Token 与主题

查找：

- CSS Variables；
- Tailwind Theme；
- SCSS/Less Variables；
- Theme Provider；
- 组件库 Token 覆盖；
- 颜色、字号、间距、圆角、阴影、层级、断点；
- 明暗主题映射。

记录每个候选项的来源和使用范围。

### 4. 公共组件

选择高频组件：

- Button；
- Input/Select；
- Form Item；
- Card；
- Table；
- Modal/Drawer；
- Tag/Badge；
- Pagination；
- Empty/Loading/Error；
- Page Header；
- Search Panel。

检查：

- Variant；
- Size；
- 默认值；
- 状态；
- 响应式行为；
- 是否被局部样式覆盖。

### 5. 代表页面

不要只看一个页面，也不要机械扫描所有页面。优先选择：

- 一个列表页；
- 一个详情页；
- 一个表单页；
- 一个仪表盘或卡片型页面；
- 一个移动端或窄屏代表页；
- 一个复杂弹窗/抽屉场景。

根据项目类型增删样本。

## 三、证据台账

每个候选规范记录：

| 字段 | 说明 |
|---|---|
| role | 设计角色，如页面边距、卡片圆角 |
| value | 实际值或组件 Variant |
| source | 文件、Token、组件或页面 |
| scope | 元素、组件、页面、公共、全局 |
| recurrence | 在多少代表页面或组件复用 |
| confidence | 高、中、低 |
| status | 已确认、候选、例外、冲突、废弃 |

示例：

```text
role: card-radius
value: 8px
source: src/styles/tokens.scss --radius-card
scope: global
recurrence: 5/5 representative pages
confidence: high
status: confirmed-by-code
```

## 四、置信度规则

### 高置信度

- 明确设计文档；
- 命名 Token；
- 主题配置；
- 公共组件正式 Variant；
- 用户已确认决策。

### 中置信度

- 多个代表页面稳定复用；
- 公共页面模板中的共享实现；
- 有一致命名但缺少正式说明。

### 低置信度

- 单个页面硬编码；
- 截图估算；
- 重复字面值但没有公共来源；
- 注释或未启用配置；
- 未被实际产品引用的示例代码。

低置信度候选不得直接写成强制规范。

## 五、冲突识别

至少检查：

- 同级标题字号不一致；
- 输入框高度存在多套；
- 页面左右边距不一致；
- 卡片圆角、边框、阴影混用；
- 同一状态颜色不同；
- 相同操作在按钮、文字链接、图标间随机变化；
- 响应式断点分散；
- z-index 任意增长；
- 同类弹窗宽度和 Footer 不一致；
- 公共组件被大量局部覆盖。

冲突不等于所有差异都错误。先判断是否存在业务、设备或密度差异。

## 六、输出分层

审计结果应分成：

1. **已确认规范**：证据充分，可直接遵循；
2. **候选规范**：模式明显，但需要用户确认；
3. **局部例外**：有合理业务原因；
4. **实现冲突**：相同角色出现无理由差异；
5. **未知项**：缺少运行环境或证据；
6. **建议决策**：需要用户决定的系统级问题。

## 七、禁止事项

- 不把出现次数最多的值自动认定为正确；
- 不从单张截图生成完整 Token 系统；
- 不复制第三方组件库全部默认 Token；
- 不把未使用代码、示例页、迁移文件作为规范证据；
- 不为了文档完整而编造缺失的颜色、间距或组件规则；
- 不在审计阶段直接大规模重构产品代码。
