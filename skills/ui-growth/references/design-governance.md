# UI 设计规范治理

本文件定义 UI Growth 如何从项目现状中提取、确认、记录和持续维护 UI 规范。

## 一、治理目标

UI 规范不是为了让所有页面完全相同，而是为了：

- 让相同角色使用相同设计语言；
- 让不同 Agent 做出一致决策；
- 降低局部样式覆盖和重复实现；
- 让新页面有明确起点；
- 让合理例外能够被解释和记录；
- 避免一次局部修改无意改变全局风格。

## 二、规范状态

每条规则必须属于以下一种状态：

### 1. Confirmed 已确认

来源于明确文档、命名 Token、正式公共组件或用户确认。

### 2. Candidate 候选

多个代表页面稳定复用，但尚未正式确认。

### 3. Exception 例外

因业务、设备、内容或交互原因有意偏离规范。

### 4. Conflict 冲突

相同角色出现无明确理由的不同实现。

### 5. Deprecated 废弃

已有替代规则，但旧页面尚未迁移。

### 6. Unknown 未知

证据不足，不能判断。

- 如何使用和更新；
- 当前适用产品和前端包。

- 控件尺寸；
- 动效原则；
- 主题模式。

- 卡片网格；

- 使用场景；

- 组件例外。

### `UI-INTERACTION-RULES.md`

- Loading；
- Empty；
- Error；
- Disabled；
- Focus；
- 删除确认；
- 提交反馈；
- 表单校验；
- 重复提交；

### `UI-RESPONSIVE-RULES.md`

- 支持设备；
- 断点来源；

- 表格策略；
- 弹窗和抽屉策略；

记录系统级决策和重要例外，避免未来 Agent 重复争论。

记录审计范围、证据、冲突、风险和建议，不把未确认建议混入正式规范。

### `page-specs/*.md`

仅用于特殊或高价值页面，记录页面专属结构、密度和例外。

## 四、规范提取流程

1. 读取现有文档；
2. 读取 Token、主题和全局样式；
3. 读取公共组件；
4. 抽样代表页面；
5. 建立证据台账；
6. 识别稳定模式和冲突；
7. 生成审计报告；
8. 对系统级候选提出决策建议；
9. 用户确认后写入正式规范；
10. 记录决策和迁移范围。

## 五、事实与建议分离

规范文档中必须区分：

- 当前项目已经如何实现；
- 推荐未来如何统一；
- 用户已经确认什么；
- 哪些旧页面尚未迁移。

错误示例：

> 项目统一使用 8px 圆角。

但实际只是多数页面使用 8px，尚未确认。

正确写法：

> 候选规范：代表页面中卡片圆角主要为 8px；另有 4px 和 12px 两类局部实现。建议将 8px 确认为默认卡片圆角，特殊营销卡片保留 12px 例外。状态：待确认。

## 六、规范变更规则

修改已确认规范时，必须说明：

- 变更原因；
- 旧规则；
- 新规则；
- 影响组件和页面；
- 是否立即迁移旧页面；
- 是否需要兼容期；
- 回退方案。

不得因为单个页面的新需求直接改写全局规范。

## 七、局部例外

例外需要满足：

1. 有明确业务或设备原因；
2. 不会导致用户把相同角色误认为不同含义；
3. 有清楚作用范围；
4. 在页面规范或决策日志中记录；
5. 不通过无说明的硬编码实现。

## 八、与代码的关系

文档是设计意图和使用合同，不应复制全部实现细节。

推荐关系：

- 设计规范说明角色和规则；
- Token/主题提供可执行值；
- 公共组件承载常用 Variant；
- 页面代码只处理局部布局和合理例外；
- 决策日志解释为什么这样做。

## 九、持续更新触发条件

以下情况需要检查是否更新文档：

- 新增全局 Token；
- 修改公共组件默认行为；
- 新增组件 Size/Variant；
- 确认新的响应式策略；
- 同类页面形成稳定新模式；
- 某个历史例外被推广为公共规则；
- 废弃旧组件或旧视觉规则。

普通局部像素调整不一定需要更新正式规范，但可以记录在页面规范或变更报告。

## 十、质量门槛

正式规范中的每条规则至少需要：

- 明确角色；
- 适用范围；
- 证据或用户确认；
- 可执行描述；
- 与现有规则不冲突；
- 必要时说明例外。

避免空泛语句，例如：

- 保持美观；
- 适当留白；
- 使用合理圆角；
- 页面要高级；
- 交互要丝滑。

这些语句必须转化为可以改变实现选择的规则。
