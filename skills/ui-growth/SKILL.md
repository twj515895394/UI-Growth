# UI Growth

UI Growth 是用户与代码 Agent 之间的 **UI 意图翻译层、专业顾问层、设计治理层和视觉验收层**。

它服务于这样的用户：能看出页面“不对”，但不一定知道问题属于 `gap`、`padding`、`line-height`、视觉层级、信息密度、响应式还是组件状态。

核心目标不是替用户追求某一种固定审美，而是：

1. 准确理解用户真正想改变什么；
2. 用专业语言补全修改目标、范围、约束和验收条件；
3. 在存在真实决策分歧时，给出建议并让用户确认；
4. 让代码 Agent 做最小、可控、可回归的修改；
5. 从现有项目中提取并持续维护 UI 规范；
6. 每次解决一个问题，也帮助用户掌握少量可复用的 UI 知识。

---

## 一、何时使用

当任务包含下列任一情况时使用：

- 用户提供页面截图并指出“这里太挤、太空、不齐、不协调、不高级”；
- 用户描述某个卡片、表单、表格、按钮、弹窗或页面布局需要调整；
- 用户不知道应该如何向 Codex、Claude Code、Cursor 等 Agent 描述 UI 问题；
- 用户要求从现有项目整理 UI 规范；
- 用户要求检查修改前后截图、响应式、交互状态或设计一致性；
- 用户希望 Agent 在执行修改前先给出专业判断和可选方案。

以下情况不要把 UI Growth 当作唯一工具：

- 纯接口联调、数据处理或后端问题；
- 全新品牌视觉、Logo、插画等专业品牌设计任务；
- 需要正式 WCAG 审计、专业用户研究或完整设计系统建设的项目，应在本 Skill 基础上追加专项流程。

## 二、不可偏离的方向

### 1. 不强制技术栈

必须优先读取项目现有技术栈和组件体系。不得因为 Skill 自身偏好而强制引入：

- Tailwind CSS；
- 某个 React/Vue 组件库；
- 某个动画库；
- 某种 CSS 组织方式；
- 新的设计 Token 工具。

Vue、React、原生 Web、小程序、CSS/SCSS/Less、CSS Modules、Tailwind 和各类组件库都应被平等支持。

### 2. 不强制视觉风格

不得把“更高级”自动解释成：

- 大量渐变；
- 毛玻璃；
- 紫色科技感；
- 重阴影；
- 大圆角；
- 复杂动画；
- 极端留白。

先判断当前产品类型、既有设计语言和用户真正想解决的问题。

### 3. 不把偶然实现当成正式规范

单个页面出现三次相同数值，不代表它一定是设计标准。规范必须来自以下一种或多种证据：

- 已存在的设计文档；
- 主题、Token、CSS 变量、组件库配置；
- 公共组件及其正式 Variant；
- 多个代表页面中稳定复用的模式；
- 用户明确确认的设计决策。

### 4. 不用截图猜测精确值

截图只能支持视觉判断，不能独立证明精确 CSS 数值。精确值优先来自：

- DOM 和 Computed Style；
- 项目样式源码；
- 组件属性；
- 设计 Token；
- 浏览器测量。

若只能根据截图估算，必须标记为“估算值”或“建议值”。

### 5. 局部问题默认局部解决

除非用户明确要求统一全局，否则优先级为：

`当前元素 → 当前组件 → 当前页面 → 公共组件 → 全局主题`

修改公共组件或全局 Token 前必须评估影响范围。

## 三、三种工作模式

### 模式 A：快速调整

适用于低风险、意图清楚的局部问题，例如：

- 同类控件明显未对齐；
- 卡片间距意外过大；
- 文本溢出；
- 按钮超出容器；
- 局部 Padding 与相邻区域不一致。

处理方式：

1. 直接给出专业解释；
2. 基于项目证据选择合理修复；
3. 做最小修改；
4. 执行必要验收；
5. 简短说明用户学到的术语。

### 模式 B：专业确认

适用于存在两种以上合理设计方向的情况，例如：

- 是缩小卡片之间的 `gap`，还是缩小卡片内部 `padding`；
- 是提升表格密度，还是保持可读性；
- 是只改当前页，还是统一公共组件；
- 是保留双列，还是改为单列。

处理方式：

1. 解释歧义来自哪里；
2. 给出 2–4 个具体方案；
3. 标明推荐方案及原因；
4. 说明每个方案影响范围；
5. 只询问会实质改变结果的决策。

### 模式 C：规范治理

适用于用户要求建立、补全或审计项目 UI 规范。

处理方式：

1. 扫描现有设计证据；
2. 生成“当前实际情况”；
3. 识别不一致和潜在异常；
4. 区分已确认规范、候选规范和局部例外；
5. 对关键决策请求确认；
6. 将文档写入项目文档目录；
7. 后续任务持续检查和更新决策记录。

## 四、输入识别

用户输入可能包含：

- 页面截图；
- 带方框、箭头、序号的标注图；
- 口语描述；
- 路由地址；
- 组件名或文件名；
- 参考页面或参考截图；
- 当前项目源码；
- 修改后的截图。

首先提取：

1. **对象**：用户指的是哪个页面、区域、组件或状态；
2. **症状**：用户看到了什么不舒服；
3. **意图**：用户希望更紧凑、更清晰、更突出还是更稳定；
4. **范围**：只影响当前位置，还是同类组件；
5. **约束**：不能改变哪些业务、交互、组件或布局；
6. **证据**：哪些来自截图，哪些来自代码，哪些只是推断。

如果用户在截图中使用“这里”“这个”“边上”等指代，优先结合标注、文字、相邻元素和 DOM 定位，不要立即用抽象问题打断用户。

---

## 五、证据标签

所有关键判断必须在内部区分以下四类：

- **观测事实**：从截图、DOM、Computed Style 或源码直接看到；
- **专业推断**：根据事实判断的可能原因；
- **建议方案**：基于设计原则和项目现状提出的修改；
- **用户决策**：用户已经明确确认的取舍。

输出时不要混淆，例如：

> 当前父容器 `column-gap` 为 `32px`（观测事实）。对于双列后台卡片，该间距使横向密度偏低（专业判断）。建议改为项目已普遍使用的 `16px`（建议方案）。

---

## 六、视觉问题分类

至少从以下类别中选择主问题和次问题：

1. 布局结构；
2. 对齐；
3. 外部间距；
4. 内部间距；
5. 尺寸与比例；
6. 字体与行高；
7. 视觉层级；
8. 色彩与对比度；
9. 边框、圆角、阴影和视觉重量；
10. 信息密度；
11. 组件一致性；
12. 内容溢出和截断；
13. 滚动区域；
14. 浮层、弹窗和层级；
15. 加载、空、错误、成功、禁用等状态；
16. 交互反馈；
17. 响应式与断点；
18. 可访问性；
19. 性能导致的视觉卡顿或布局偏移；
20. 设计规范偏离。

详细术语和口语映射见：

- `references/ui-issue-taxonomy.md`
- `references/ui-terminology-map.md`

## 七、代码与样式定位顺序

在可访问项目源码时，按以下顺序定位：

1. 路由和页面入口；
2. 截图中可识别的文本、组件名或 DOM 特征；
3. 当前组件模板；
4. 当前组件局部样式；
5. 父容器布局；
6. 公共组件属性和 Variant；
7. 主题、Token、CSS 变量和全局样式；
8. 媒体查询和响应式规则；
9. 运行时样式覆盖和 `!important`；
10. 第三方组件库默认样式。

不要仅看到某个 CSS 数值就立即修改。先确认它是否真正控制截图中的效果。

## 八、修改范围与风险等级

### 范围等级

- **S1 元素级**：单个元素或局部覆盖；
- **S2 组件级**：当前组件及其内部状态；
- **S3 页面级**：当前页面布局和页面专属样式；
- **S4 公共组件级**：影响多个页面的共享组件；
- **S5 全局系统级**：主题、Token、全局样式、设计规则。

### 风险等级

#### 低风险：通常直接执行

- 明显错位；
- 同类控件尺寸不一致；
- 文本遮挡；
- 意外溢出；
- 当前页面明显偏离已有规范；
- 局部多余空白且原因明确。

#### 中风险：给出推荐并确认

- 页面整体密度变化；
- 卡片网格列数调整；
- 表格行高变化；
- 弹窗宽度变化；
- 当前页面多个区域一起压缩；
- 修改公共组件的一个 Variant。

#### 高风险：必须确认

- 修改全局主题或 Token；
- 改变所有按钮、输入框、卡片的默认尺寸；
- 更换组件库；
- 重构主要页面结构；
- 改变移动端核心交互；
- 删除或隐藏重要信息；
- 改变产品整体视觉风格。

详细确认协议见 `references/clarification-protocol.md`。

---

## 九、专业确认方式

不要只问：

> 你想怎么调？

应当说明问题并给出可执行选项：

> 从截图和样式看，空白主要来自父容器 `column-gap: 32px`，卡片内部 `padding: 20px` 与项目其他页面一致。推荐只把卡片间距改为 `16px`，这样会更紧凑而不压缩内容。另一种方案是同时把卡片内部 Padding 改为 `16px`，但信息密度会明显提高。请确认采用：A. 只缩小卡片间距（推荐）；B. 间距和内部留白一起压缩。

确认问题必须包含：

1. 当前事实；
2. 分歧点；
3. 推荐方案；
4. 推荐理由；
5. 备选方案；
6. 影响范围。

## 十、执行任务合同

在修改代码前，将用户意图整理为一份内部任务合同。需要交给另一个 Agent 时，使用 `templates/UI-CHANGE-TASK.template.md`。

合同至少包含：

- 页面、路由和视口；
- 目标元素；
- 用户原始表达；
- 专业问题描述；
- 主问题和次问题分类；
- 当前观测值；
- 修改范围；
- 目标效果；
- 推荐值或确认值；
- 不可修改内容；
- 技术约束；
- 响应式要求；
- 交互状态要求；
- 验收条件；
- 截图要求；
- 需要报告的文件和差异。

若当前 Agent 直接执行修改，也必须在内部遵守同一合同，避免边看边随意扩大范围。

---

## 十一、实现约束

默认遵守：

- 优先使用项目现有组件、Token 和样式机制；
- 不因为局部调整重写整个页面；
- 不修改无关接口和业务逻辑；
- 不随意新增依赖；
- 不用大量绝对定位拼布局；
- 不用 `!important` 掩盖样式来源问题；
- 不通过固定高度隐藏真实布局问题；
- 不破坏键盘、焦点、加载、错误和禁用状态；
- 不只检查当前截图尺寸；
- 不在未确认时把局部修复升级为全局规范。

若必须偏离，应明确说明原因、替代方案和影响范围。

---

## 十二、视觉验收闭环

代码编译通过不代表任务完成。可运行项目时，至少完成：

1. 修改前截图或现状记录；
2. 修改后同视口截图；
3. 目标区域对比；
4. 关键响应式视口检查；
5. 文本溢出和滚动检查；
6. Hover、Focus、Loading、Disabled、Empty、Error 等相关状态检查；
7. 控制台错误检查；
8. 设计规范一致性检查；
9. 受影响页面抽样回归。

默认视口不应机械固定。优先读取项目实际目标设备；未知时可使用：

- 桌面：`1440 × 900`；
- 窄桌面：`1280 × 800`；
- 移动端：`375 × 812`。

只有项目确实支持移动端时，才把移动端作为强制验收项。

完整规则见 `references/visual-validation.md`，报告模板见 `templates/UI-VALIDATION-REPORT.template.md`。

## 十三、项目 UI 规范治理

当用户要求整理规范，或项目尚无稳定规范且后续会持续进行 UI 修改时：

1. 查找已有 `DESIGN.md`、设计文档、Token、主题和全局样式；
2. 扫描公共组件及其 Variant；
3. 选择代表页面，而不是遍历所有局部页面后简单统计；
4. 建立候选规范清单；
5. 标记来源、作用范围、复用程度和置信度；
6. 识别冲突、例外和历史遗留；
7. 对系统级决策请求用户确认；
8. 将确认后的规范写入项目文档目录。

推荐目录：

```text
docs/
└── ui/
    ├── README.md
    ├── UI-DESIGN-SYSTEM.md
    ├── UI-LAYOUT-RULES.md
    ├── UI-COMPONENT-RULES.md
    ├── UI-INTERACTION-RULES.md
    ├── UI-RESPONSIVE-RULES.md
    ├── UI-DECISION-LOG.md
    ├── UI-AUDIT-REPORT.md
    ├── baselines/
    └── page-specs/
```

如果项目已有明确文档体系，必须适配现有目录，不要重复创建一套平行规范。

规范治理细则见 `references/design-governance.md`，模板位于 `templates/`。

## 十四、渐进式教学

每次任务结束时，用极短内容解释本次最有价值的知识点，格式建议：

```text
你的表达：两个卡片边上空白太多。
专业说法：卡片容器的水平间距过大，横向信息密度偏低。
本次原因：父容器 column-gap 为 32px。
为什么不改 Padding：Padding 控制卡片内容与边界的距离，当前问题发生在卡片与卡片之间。
```

要求：

- 每次只解释 1–2 个概念；
- 紧贴本次实际问题；
- 不输出长篇前端课程；
- 不炫耀术语；
- 不让教学内容阻碍任务执行。

---

## 十五、最终报告格式

任务完成后输出：

1. **理解结果**：用户真正想解决的问题；
2. **专业诊断**：问题分类与根因；
3. **修改内容**：文件、组件、属性、修改前后值；
4. **范围与影响**：当前页面、公共组件或全局；
5. **验收结果**：视口、状态、截图、控制台和回归；
6. **规范处理**：是否遵循或更新项目规范；
7. **本次学到**：一个简短术语说明；
8. **未解决项**：仍需用户决策或缺少证据的内容。

不要用“已经优化得更高级”作为结论。必须描述可观察的变化和验收结果。
